# 白金英雄坛说 - Godot 技术架构

> 版本：v3.0
> 更新日期：2026-05-31
> 适用工程：`godot_project/`，Godot 4.6 / GDScript

---

## 1. 工程边界

当前仓库只保留一个可运行游戏工程：

```text
yxts-llm/
├── godot_project/          # Godot 主工程
│   ├── project.godot
│   ├── scenes/
│   ├── scripts/
│   ├── data/
│   └── assets/
├── docs/                   # 设计文档与审计
├── tools/                  # 资源生成和数据校验
└── Godot_v4/               # 本地 Godot 引擎目录，git 忽略
```

旧 Python/Pygame 原型、早期 Godot 草稿目录和未接入的旧素材源已经清理。后续实现、校验和导出都以 `godot_project/` 为准。

## 2. 运行入口

- `godot_project/project.godot`：Godot 项目配置。
- `godot_project/scenes/main.tscn`：主场景。
- `godot_project/scripts/main.gd`：主控制器，负责装配世界、玩家、UI、战斗和氛围层。

运行命令：

```bash
./Godot_v4/linux/Godot_v4.6.3-stable_linux.x86_64 --path godot_project
```

头less 验证命令：

```bash
env HOME=/tmp/godot_home XDG_DATA_HOME=/tmp/godot_data XDG_CONFIG_HOME=/tmp/godot_config XDG_CACHE_HOME=/tmp/godot_cache ./Godot_v4/linux/Godot_v4.6.3-stable_linux.x86_64 --headless --path godot_project --quit-after 1
```

## 3. Autoload

当前 Godot 工程使用四个核心 Autoload：

| 单例 | 文件 | 职责 |
|---|---|---|
| `GameData` | `scripts/autoload/game_data.gd` | 加载 NPC、区域、物品、任务、资源映射和静态文本 |
| `GameState` | `scripts/autoload/game_state.gd` | 保存玩家状态、任务、背包、装备、探索度、江湖传闻、时间天气和存档 |
| `EventBus` | `scripts/autoload/event_bus.gd` | 跨 UI、地图、战斗、任务、世界事件的信号分发 |
| `LLMDirector` | `scripts/autoload/llm_director.gd` | NPC 动态闲谈、世界上下文 prompt 组装和可选 LLM 后端请求 |

短期内 `GameState` 承担了较多规则逻辑。后续应把背包、任务、修炼、经济、声望拆为独立系统，`GameState` 只保留状态快照和存档序列化。

| `npc_sprite_assets.json` | NPC 名称到地图 sprite 的映射 |
| `npc_portrait_assets.json` | NPC 名称到对话头像的映射 |
| `item_icon_assets.json` | 物品 ID 到图标的映射 |

## 5. 场景与脚本分层

```text
scripts/
├── autoload/       # 全局数据、状态、事件
├── entities/       # PlayerActor / NpcActor
├── systems/        # 战斗等规则系统
├── ui/             # HUD、小地图、对话、背包、商店、任务、修炼、世界地图、战斗面板
├── world/          # 大地图生成、局部区域/商铺内景、区域识别、NPC 生成、可通行判定
└── main.gd         # 主装配与流程协调
```

当前核心流程：

1. `main.gd` 初始化 `GameData` 和 `GameState`。
2. `WorldMap` 根据 `regions.json` 和程序化地貌规则生成连续世界地图。
3. 玩家按 `E` 可从世界地图进入 `LocalAreaMap` 局部区域，局部区域再进入商铺内景或返回世界地图。
4. `PlayerActor` 负责输入移动，移动后更新当前区域和探索度。
5. `WorldMap` / `LocalAreaMap` 根据当前活动地图生成 NPC 节点，并提供附近交互和入口查询。
6. UI 面板通过 `EventBus` 与 `GameState`、`CombatSystem`、当前活动地图通信。
7. 任务完成和首次发现区域会写入 `GameState.world_events`，任务日志、NPC 闲谈和 LLM prompt 会读取最近江湖传闻。
8. 存档由 `GameState` 序列化为 JSON；在局部地图和商铺内景中快速存档时保存世界地图返回点，并保存江湖传闻。

## 7. LLM 接入

`LLMDirector` 默认不依赖网络，会根据 NPC 身份、性格、关系、记忆、当前区域、天气、任务状态和最近江湖传闻生成离线动态闲谈。这样没有 API Key 时游戏仍可运行。

如需接入真实大模型，可在启动 Godot 前设置以下环境变量：

```bash
export YXT_LLM_ENDPOINT="http://127.0.0.1:8000/v1/chat/completions"
export YXT_LLM_API_KEY="your-api-key"
export YXT_LLM_MODEL="gpt-4.1-mini"
```

后端需兼容 OpenAI Chat Completions 响应结构。`DialoguePanel` 会先显示离线世界导演台词；如果后端返回成功，再用模型生成台词替换当前对话文本。

## 8. 当前技术债

- `GameState` 仍然过重，需要拆出背包、任务、修炼、经济、声望系统。
- 世界地图还不是正式 TileMapLayer/TileSet，地形拼接和碰撞仍依赖脚本规则。
- 战斗系统仍是竖切版本，缺少完整招式、状态、AI、浮字和技能表现。
- `LLMDirector` 已有本地回退和 HTTP 接口，但还没有流式输出、对话历史管理、NPC 长期记忆摘要和安全过滤。
- UI 已可用，但还没有完整皮肤、动效、响应式缩放和键位配置。
- 自动化测试仍不足，目前主要依赖数据校验脚本和 Godot 头less启动验证。

## 9. 开发原则

1. `godot_project/data/` 是运行数据的单一来源。
2. 新功能优先走 Godot 原生场景、脚本、资源映射，不重新引入旧 Python 运行链路。
3. 每次扩展 NPC、区域、物品、任务或美术后，都运行 `tools/validate_godot_data.py`。
4. 大面积改动后再跑 Godot 头less启动，确认资源路径和场景装配没有断。
5. 文档只记录当前工程或明确的未来目标，避免保留已经废弃的旧路径。
